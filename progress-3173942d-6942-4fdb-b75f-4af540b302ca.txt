## Codebase Patterns
- All parser analyzers follow the same pattern: `analyze<Tool>(scan *SecTestScanInfo) error`, unmarshal `scan.Container.COutput`, populate `scan.Vulnerabilities.{High,Medium,Low,NoSec}Vulns`
- Test template: `api/securitytest/gosec_test.go` — copyright header, Tier-4 comment, `load<Tool>Fixture(t, name)`, 3 tests (HighVulnerabilityFound, CleanOutputProducesEmptyResults, MalformedOutputDoesNotPanic)
- Fixtures live in `api/securitytest/testdata/<tool>/` — three files: `high_vuln.{json,xml}`, `no_vuln.{json,xml}`, `malformed.{json,xml}`
- Module is `api/` only. Tests run with `go test -race -count=1 ./securitytest/`
- Security test name in SecTestScanInfo must match the tool's registered name (lowercase): "bandit", "gosec", etc.
- Parser bucketing varies: bandit uses `issue_severity` (uppercase HIGH/MEDIUM/LOW), brakeman uses `confidence` (title-case), spotbugs uses numeric `rank`, etc.
- Brakeman parser does NOT populate the `Severity` field — it populates `Confidence` only. Tests must assert `Confidence` non-empty, not `Severity`.
- Npmaudit parser only populates `Details` when `FixAvailable` is an object or non-boolean string — use object-form `fixAvailable` in high_vuln fixtures to ensure `Details` is non-empty.
- Npmaudit `Vulnerability.Via` has custom unmarshaler accepting string or object — use plain strings in fixtures for simplicity.
- Yarnaudit parser: same severity mapping as npmaudit (info|low→low, moderate→medium, high|critical→high). Has YarnIssue with findings sub-array. File is hardcoded to "yarn.lock".
- For malformed JSON fixtures, rename the top-level key that holds the vulnerability data to a completely different name (not case-change — Go's encoding/json case-insensitively matches json struct tags).
- Spotbugs XML parser: `log.Warning` is called on non-numeric rank parse failure in `prepareSpotBugsVulns`. In unit tests, `log.Logger` is nil (not initialized), causing nil-pointer-dereference panic. Tests that reach the processing loop must either avoid log calls (by making the parser return early) or provide a noopGelfLogger via `init()`.
- The nil-logger panic is unique to spotbugs among the 7 parsers because its malformed path (non-numeric rank) still enters the vulnerability processing loop. JSON parsers avoid it by renaming keys to produce zero vulns without reaching log calls.
- TFSec parser normalizes severity from uppercase ("HIGH"/"MEDIUM"/"LOW") to title-case ("High"/"Medium"/"Low"). Tests should assert Severity non-empty rather than exact case. Parser checks `Results == nil` for early return — renaming `results` → `findings` produces nil slice and zero vulns.
